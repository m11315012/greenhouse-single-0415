# D-Link DAP-3662 403 Forbidden 與 xmldb Socket 問題排查與修復

**韌體 SHA256**：`b000df79be7475fe9b6f39e532a54f9034dcd66bc105a21a1c3b3f911b1d20ab`  
**平台**：Greenhouse（FirmAE）+ QEMU MIPS big-endian user-space emulation  

---

## 背景

在用 Greenhouse 對 D-Link DAP-3662 REVA 韌體做模擬與安全研究時，韌體成功跑起來後，用瀏覽器打開 Web 介面，所有 PHP 頁面都回傳 **403 Forbidden**，完全無法進入管理介面。

---

## 第一階段：定位 httpd 是誰

首先確認 Web Server 是哪個程式。查看 Greenhouse 生成的 `qemu_run.sh`：

```sh
/sbin/httpd -n -s wapac11_dkbs_dap3662 -f /var/etc/httpd.cfg
```

確認是 `/sbin/httpd`，然後把它從 Docker 容器裡拉出來：

```bash
docker cp debug_gh_rehosted_1:/fs/sbin/httpd /tmp/httpd_original
file /tmp/httpd_original
```

結果是 **MIPS big-endian ELF**，不是標準的 lighttpd/boa/goahead，而是 D-Link 自家基於 **mathopd** 改造的 `alphanetworks httpd`。

---

## 第二階段：搜尋 403 的來源

### 2.1 先試 curl，確認問題行為

```bash
curl -v http://<QEMU_IP>/session_login.php
```

回傳 403，Header 裡沒有任何 `Set-Cookie`，也沒有 `WWW-Authenticate`。這說明 403 **不是密碼錯誤**，而是在認證之前就被擋住了。

### 2.2 用 strings 找特徵字串

```bash
strings /tmp/httpd_original | grep -i session
strings /tmp/httpd_original | grep -i auth
```

找到兩個關鍵字串：

- `parse_auth_key` — alphanetworks httpd 的特有字串
- `r->keep_key` — session key 相關的結構體存取

這確認了這是 alphanetworks 系列的 httpd。

### 2.3 用 objdump 反組譯

```bash
objdump -d /tmp/httpd_original > /tmp/httpd.asm
```

搜尋 `parse_auth_key` 字串的 xref，找到一個函式叫 `check_session`，位於虛擬位址 **`0x40e09c`**。

## 第三階段：追蹤 403 的觸發路徑

### 3.1 錯誤假設：以為 parse_auth_key 區段就是主路徑

一開始以為 `parse_auth_key` 附近的程式碼就是 403 的主要來源，在那附近找到了兩個 branch 指令：

- `0x0e3c0`：`beqz` — key 長度檢查
- `0x0e3e0`：`beqz` — 另一個條件跳轉

對這兩個位置打 patch，把它們 NOP 掉或改成無條件跳轉，重新部署測試。

**結果：還是 403。**

### 3.2 重新思考：這段 code 根本沒被走到

仔細看 `parse_auth_key` 附近的 xref，發現這個函式只有從 `0x40e4a8` 被呼叫，而 `0x40e4a8` 的上下文是 `logout.php` 的處理路徑，也就是說：

> 我們剛才 patch 的地方，是**登出流程**的 code，根本不是一般 PHP 請求的 403 來源。

### 3.3 往上追：找 process_request 主函式

繼續往上找，定位到主要的 HTTP 請求處理函式 `process_request@@Base`，位於 **`0x410f08`**。

在這個函式裡，用二進位搜尋找呼叫 session check 之前的 guard 條件：

```python
data = open('/tmp/httpd_original','rb').read()
# 搜尋 BLEZ +4 / NOP / JAL 的組合
pattern = bytes([0x1a,0x00,0x00,0x04, 0x00,0x00,0x00,0x00, 0x0c])
idx = 0
while True:
    pos = data.find(pattern, idx)
    if pos == -1: break
    print(hex(pos))
    idx = pos + 1
```

找到位於檔案偏移 **`0x12ac0`**（虛擬位址 `0x412ac0`）的位置。

## 第四階段：理解這段組合語言

用 objdump 看 `0x412ac0` 附近：

```asm
412ac0: 1a000004  blez  $s0, 412ad4   ← 關鍵：如果 s0 <= 0，跳過 session check
412ac4: 00000000  nop
412ac8: 0c103827  jal   40e09c        ← 呼叫 session 驗證函式
412acc: 02602021  move  $a0, $s3      ← delay slot：傳入 request struct
412ad4: 8e620ac0  lw    $v0, 2752($s3) ← 檢查 403 flag 有沒有被設定
412adc: 1440000a  bnez  $v0, 412b08   ← 如果 403，跳去處理錯誤
```

### 邏輯解釋

- `$s0` 存的是「這個請求需不需要做 session 驗證」的計數或 flag
- `blez $s0, 412ad4`：**如果 s0 ≤ 0，就跳過 session check，直接繼續**
- 但如果 s0 > 0（需要驗證），就跑 `jal 40e09c` 做 session 驗證
- 驗證失敗 → 設 403 flag → 回傳 403

在 Greenhouse 模擬環境下，session 系統沒有正確初始化，驗證永遠失敗，所以只要走到 session check 就一定 403。

## 第五階段：設計 Patch

### MIPS 指令格式說明

MIPS I-type branch 指令格式（32 bit）：

```
[31:26] opcode  [25:21] rs  [20:16] rt  [15:0] offset
```

| 指令 | opcode | 第一 byte（以 $s0 為 rs） |
|------|--------|--------------------------|
| `BLEZ $s0, +4` | `0x06` | `0x1a` |
| `B +4`（= `BEQ $zero,$zero,+4`） | `0x04` | `0x10` |

只要把 `0x12ac0` 位置的 byte 從 `0x1a` 改成 `0x10`，`BLEZ` 就變成 `B`（永遠跳轉），永遠跳過 session check。

### 實際 patch 腳本

```python
patches = [
    # 主要 patch：BLEZ $s0,+4 → B +4（永遠跳過 session check）
    (0x12ac0, b'\x1a\x00\x00\x04', b'\x10\x00\x00\x04'),
    # 輔助 patch（雖在錯誤路徑，保留無害）
    (0x0e3c0, b'\x16\x02\x00\x09', b'\x00\x00\x00\x00'),
    (0x0e3e0, b'\x10\x40\x00\x1f', b'\x10\x00\x00\x1f'),
]

data = bytearray(open('/tmp/httpd_original', 'rb').read())
for offset, old, new in patches:
    assert data[offset:offset+len(old)] == old, f"Mismatch at {hex(offset)}"
    data[offset:offset+len(new)] = new

open('/tmp/httpd_patched2', 'wb').write(bytes(data))
```

## 第七階段：xmldb socket 殘留問題

在重啟 Docker container 後，發現 `index.php` 回傳 500，進去查：

```bash
docker exec debug_gh_rehosted_1 ls -la /fs/var/run/xmldb_sock
```

發現 `/fs/var/run/xmldb_sock` 這個 socket 檔案**還存在**，但 xmldb process 已經不在了。

### 問題原因

xmldb daemon 啟動時會嘗試 `bind()` 這個 socket。如果 socket 檔案已存在，`bind()` 失敗，xmldb 直接退出，導致整個 XML 設定資料庫無法運作，PHP 頁面因讀不到設定而 500。

### 手動修復驗證

```bash
docker exec debug_gh_rehosted_1 \
    rm -f /fs/var/run/xmldb_sock /fs/var/run/xmldb_sock_*.pid
# 重新啟動 xmldb → index.php 回傳 200 ✓
```

## 第八階段：整合進 Greenhouse 自動修復邏輯

在 xmldb 啟動命令之前插入 `rm -f`，讓 Greenhouse 生成的 `run_background.sh` 每次都先清理舊 socket：

### Fix 2 — alphanetworks httpd session bypass patch

**檔案**：`Greenhouse/backend/Planter.py`，`Fixer` class

#### 在 `initial_setup()` 末尾新增呼叫（第 117 行）

```python
self.propgate_contents(fs_path)

```

#### 新增方法 `patch_alphanetworks_httpd()`

```python
def patch_alphanetworks_httpd(self, fs_path):
    # 偵測 alphanetworks httpd 並 patch session check guard
    # BLEZ $s0,+4 → B +4，讓所有請求跳過 session 驗證，避免 403
    SIGS = [b"parse_auth_key", b"r->keep_key"]
    PATTERN = bytes([0x1a, 0x00, 0x00, 0x04, 0x00, 0x00, 0x00, 0x00, 0x0c])
    for root, dirs, files in os.walk(fs_path):
        for fname in files:
            fpath = os.path.join(root, fname)
            if not os.path.isfile(fpath) or os.path.islink(fpath):
                continue
            try:
                with open(fpath, 'rb') as f:
                    data = bytearray(f.read())
            except Exception:
                continue
            if len(data) < 4 or data[:4] != b'\x7fELF':
                continue
            if not all(sig in data for sig in SIGS):
                continue
            print("    - [httpd_bypass] Detected alphanetworks httpd: %s" % fpath)
            idx = 0
            patched = False
            while True:
                pos = bytes(data).find(PATTERN, idx)
                if pos == -1:
                    break
                if pos % 4 == 0:
                    data[pos] = 0x10  # BLEZ (0x1a) → B (0x10)
                    print("    - [httpd_bypass] Patched BLEZ→B at offset 0x%x" % pos)
                    patched = True
                idx = pos + 1
            if patched:
                try:
                    with open(fpath, 'wb') as f:
                        f.write(data)
                except Exception as e:
                    print("    - [httpd_bypass] ERROR writing patched binary: %s" % e)
```

#### 偵測邏輯說明

| 步驟 | 說明 |
|------|------|
| Walk filesystem | 遍歷整個 extracted fs |
| ELF 過濾 | 跳過非 ELF、symlink、非一般檔案 |
| 字串簽名比對 | 同時含 `parse_auth_key` + `r->keep_key` → 確認是 alphanetworks httpd |
| Pattern 搜尋 | 找 `1a 00 00 04 00 00 00 00 0c`（BLEZ+4 / NOP / JAL 開頭），且在 4-byte 對齊位置 |
| Patch | byte `0x1a` → `0x10` |
| 寫回 | 覆寫原檔案 |

## 總結

| 問題 | 根本原因 | 修復位置 | 修復方式 |
|------|----------|----------|----------|
| 所有 PHP 頁面 403 | alphanetworks httpd 在 `process_request` 中強制做 session 驗證，Greenhouse 環境下 session 未初始化，永遠失敗 | `Greenhouse/backend/Planter.py` | `BLEZ $s0,+4`（`0x1a`）→ `B +4`（`0x10`），永遠跳過 session check |
| 重啟後 index.php 500 | xmldb 重啟時找到舊 socket 檔案，`bind()` 失敗，daemon 退出，PHP 無法取得設定 | `Greenhouse/sidebins/xmldb.py` | xmldb 啟動前加 `rm -f` 清理舊 socket |

兩個修復已整合進 Greenhouse 的自動修復流程，之後重新 run Greenhouse 對 alphanetworks 系列韌體時會自動處理，不需要手動介入。
